# 17 · 发布与上架（Publish & Release）

> 写完代码只是第一步，真正上线要走一条**固定流水线**：上传代码 → 生成开发版 → 设体验版测试 → 提交审核 → 审核通过 → 发布上线。审核是**人工+机器双重把关**，最容易在这一步被卡。重要度 ⭐⭐。官方：developers.weixin.qq.com

## 🎯 一句话核心

小程序不像 H5"传服务器就上线"，而是**提审上架**制：你在**微信开发者工具**里点"上传"，代码进入微信后台成为**开发版**；把它设为**体验版**给测试同学扫码试用；没问题后**提交审核**，微信审核团队人工审 1~7 天；**审核通过**后你手动点**发布**，用户才能搜到线上版。四个版本（开发/体验/审核/线上）本质是**同一份代码在流水线不同阶段的状态**。

## 📖 核心概念 / 讲解

**★ 发布流程（重点，务必背下这条流水线）。** 从写完代码到用户能用，标准路径如下：

```
                     ┌──────────── 微信开发者工具（本地）────────────┐
                     │  写代码 → 真机预览自测 → 点【上传】            │
                     │  （填 版本号 version + 项目备注 desc）        │
                     └───────────────────────┬──────────────────────┘
                                              │ 上传代码
                                              ▼
              ┌──────────────── 微信公众平台后台（web 管理端）────────────────┐
              │                                                              │
              │   ①【开发版】 ← 每次上传覆盖，只有一个                        │
              │        │ 点"设为体验版"                                       │
              │        ▼                                                     │
              │   ②【体验版】 → 生成体验二维码，邀请"体验成员"扫码测试        │
              │        │ 点"提交审核"（填类目/功能页面/测试账号）             │
              │        ▼                                                     │
              │   ③【审核版】 → 微信团队人工+机器审核（约 1~7 天）           │
              │        │                                                     │
              │    ┌───┴───┐                                                 │
              │  驳回      通过                                              │
              │  （改）      │ 点"发布"（全量 / 分阶段灰度）                 │
              │              ▼                                               │
              │   ④【线上版】 → 所有微信用户可搜索/扫码访问 ✅              │
              └──────────────────────────────────────────────────────────────┘
```

关键动作都在两个地方发生：**开发者工具**只负责"上传代码"，其余（设体验版、提审、发布、回滚）全在 **微信公众平台后台（mp.weixin.qq.com）** 点。上传时**必填版本号 + 项目备注**，备注写清本次改了啥，后台会按版本号列出历史，方便回滚。

**版本概念（同一份代码的四个状态）。** 这是新手最容易绕晕的点，对照记：

| 版本 | 谁产生 | 几个 | 谁能访问 | 作用 |
|---|---|---|---|---|
| **开发版**（Dev） | 开发者工具**上传**后自动生成 | 只有 1 个（新上传覆盖旧的） | 无（只是后台一条记录） | 上传后的原始产物，待处理 |
| **体验版**（Trial/Experience） | 由开发版**手动设为**体验版 | 只有 1 个 | **体验成员**（后台加白名单）扫二维码 | 内测，提审前最后验证 |
| **审核版** | **提交审核**后进入 | 1 个 | 微信审核团队 | 官方审核中的快照 |
| **线上版**（Release） | 审核通过后**点发布** | 1 个 | **全体微信用户** | 正式对外服务的版本 |

> 对比 Web 发布：H5 是 `git push` → CI 构建 → 部署，几分钟自己说了算；小程序多了**人工审核**这道关，且发布节奏受平台约束——这是"提审上架"和"部署 URL"的本质差别（见 [01-overview-architecture](../01-overview-architecture/)）。

**★ 常见拒审原因（实战血泪，提审前逐条自查）。** 审核被驳回极常见，会附一条驳回理由。高频原因：

- **类目与内容不符**：注册时选的**服务类目**（如"工具"）和小程序实际内容（其实在卖货）对不上 → 按真实业务重选类目。
- **涉及未授权/未开通的功能**：用了支付、直播、获取手机号等需**单独申请开通**的接口却没开权限；或调用了当前主体资质不允许的能力。
- **诱导分享 / 诱导关注**："分享给 3 个好友解锁""转发领红包"等强制/利诱行为，明确禁止。
- **信息不完整**：功能页面填得太少、测试路径走不通、缺少必要的说明文案。
- **有 bug / 白屏 / 报错 / 功能跑不通**：审核员真机点击时闪退、白屏、按钮无反应、死链 → 提审前必须真机全流程走一遍。
- **需资质的行业没资质**：医疗、金融、教育、直播、征信等**特殊行业**需上传对应**资质证明**（如医疗器械经营许可、ICP 经营许可、金融牌照），无资质直接拒。
- **缺用户隐私协议 / 隐私声明**：收集用户信息（手机号、位置、相册等）却没配置**《小程序用户隐私保护指引》**、没有弹窗告知授权用途 → 现为强制项，不配会被拒甚至下架。
- **测试账号未提供**：小程序需登录才能用，却没在提审表单里给**审核专用测试账号/验证码**，审核员进不去 → 一定要在"审核备注"里填可用测试账号。

**上线前检查（提审前的体检）。**

- **合法域名（业务域名 / request 合法域名）**：所有 `wx.request`、上传下载、WebSocket 的域名必须提前在后台"开发管理 → 服务器域名"里配置且**已备案 HTTPS**，否则线上请求全部失败（开发工具里"不校验域名"只在开发时有效，见 [09-network-storage](../09-network-storage/)）。
- **真机测试**：不同机型、iOS/Android、弱网下真机跑核心链路，模拟器过 ≠ 真机过。
- **体验评分 / 性能**：用开发者工具的"体验评分"跑一遍，处理白屏时间、setData 过大等警告（见 [13-subpackage-performance](../13-subpackage-performance/)）。
- **隐私协议 / 用户信息收集声明**：配置隐私保护指引，凡 `getUserProfile`、`getPhoneNumber`、`chooseLocation` 等收集行为都要声明用途（见 [11-login-auth](../11-login-auth/)）。
- **★ ICP 备案（2023 新规，强制）**：自 2023 年起新上架小程序须完成**工信部 ICP 备案**，未备案无法提审/上架。在微信后台"设置 → 基本设置 → 小程序备案"提交主体与负责人信息，走管局审核（约数天到数周）。**务必预留备案时间**，别等提审当天才发现没备案。

**版本管理与回滚。** 后台"版本管理"里保留历史线上版本，若新版本上线后发现严重问题，可一键**回退（回滚）到指定历史版本**，用户端会恢复到旧版——这是线上事故的第一急救手段。

**运营（上线不是终点）。**

- **数据分析**：后台"统计"提供访问量、留存、访问来源、页面路径、实时访客等；也可接入自定义埋点。
- **灰度 / 分阶段发布**：发布时可选**分阶段发布**（如先放量 10% → 50% → 100%），出问题及时"停止灰度"止损。
- **A/B 测试**：对不同用户群投放不同版本/文案，用数据决策。

**抖音等其它平台（流程类似，各审各的）。** 抖音/字节小程序在**抖音开放平台（微信换成字节的后台）** 走几乎相同的流水线：上传 → 体验版 → 提审 → 发布；同样有类目、资质、隐私、备案要求，只是后台入口、审核团队、部分规则不同（tt API 差异见 [15-douyin](../15-douyin/)）。跨端项目（uni-app/Taro，见 [16-cross-platform](../16-cross-platform/)）一套代码要**在各平台后台分别提审**，各平台独立审核、独立上线。

## 💻 代码示例：上传相关配置与命令行发布

上传本身是在开发者工具点按钮，但有几处配置和 CI 方式值得知道：

```jsonc
// project.config.json：与上传/发布相关的部分设置
{
  "appid": "wx1234567890abcdef",   // 你的小程序 AppID（决定传到哪个小程序）
  "projectname": "my-miniprogram",
  "setting": {
    "urlCheck": true,      // 上传前建议开启域名校验，避免线上请求挂掉
    "es6": true,           // 转 ES6，提审代码要经过编译
    "minified": true       // 压缩代码，减小包体积（主包 ≤ 2MB 限制）
  }
}
```

```bash
# 用官方 miniprogram-ci 在 CI/CD 里自动上传（替代手点"上传"，适合团队）
npm i miniprogram-ci
```

```js
// upload.js：命令行上传为"开发版"（需在后台配置上传密钥 + IP 白名单）
const ci = require('miniprogram-ci');

(async () => {
  const project = new ci.Project({
    appid: 'wx1234567890abcdef',
    type: 'miniProgram',
    projectPath: './dist',                 // 代码目录
    privateKeyPath: './private.key',       // 后台下载的上传密钥
    ignores: ['node_modules/**/*'],
  });

  await ci.upload({
    project,
    version: '1.2.0',                      // ★ 版本号（必填，对应后台版本）
    desc: '修复登录白屏 + 新增订单页',      // ★ 项目备注（必填，写清改动）
    setting: { es6: true, minified: true },
  });
  console.log('上传成功 → 去后台设体验版/提审');
})();
// 注意：CI 只能传到"开发版"；设体验版、提审、发布仍需在后台操作/调审核 API
```

## 🔑 要点速记

- **发布流水线**：写码 → 开发者工具**上传**(填版本号+备注) → 后台**开发版** → 设**体验版**(邀体验成员测) → **提交审核**(1~7天) → 通过后**发布**(可灰度) → **线上版**。
- **四版本**=同一份代码四状态：**开发版**(上传产物)/**体验版**(内测)/**审核版**(审核中)/**线上版**(全量用户)。
- 上传只在**开发者工具**；设体验版、提审、发布、回滚都在**微信公众平台后台**。
- **★ 高频拒审**：类目不符、用未授权功能、诱导分享、信息不全、有bug/白屏、特殊行业无资质、缺隐私协议、没给测试账号。
- **上线前必查**：合法域名(已备案HTTPS)、真机全链路、体验评分、隐私声明、**ICP 备案**(2023新规强制，预留时间)。
- 出事可**回滚**到历史线上版本；运营看**数据统计**、用**分阶段灰度**发布止损。
- **抖音/其它平台**流程类似，各自后台各自提审；跨端项目要**逐平台分别提审**。

## ⚠️ 易错点 / 最佳实践

- ⚠️ **别忘配"服务器域名"**——开发工具勾了"不校验合法域名"能跑，一上线真机全部请求失败；提审前务必在后台配好并确认域名已备案 HTTPS。
- ⚠️ **提审必给测试账号**——需登录的小程序，审核备注里不填可用账号/验证码，审核员进不去必被拒。
- ⚠️ **类目要选对**——类目与实际功能不符是头号驳回原因；特殊行业(医疗/金融/教育)提前备好资质。
- ⚠️ **ICP 备案要趁早**——2023 新规下没备案无法提审，备案审批要数天到数周，别卡在 deadline。
- ⚠️ **诱导分享是红线**——"转发解锁""邀请得奖励"等强制/利诱裂变会被拒，甚至封禁。
- ✅ **提审前真机走全流程**——白屏、闪退、死链是常见拒审点，模拟器过不代表真机过。
- ✅ **首发用分阶段灰度**——先放小比例流量观察崩溃率/数据，稳了再全量，出问题立即停止灰度或回滚。
- ✅ **版本号 + 备注写规范**——便于后台追溯与回滚；团队用 `miniprogram-ci` 把上传接入 CI/CD。
